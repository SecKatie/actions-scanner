[[overview]]
== Overview

CAUTION: This tool identifies a critical security vulnerability. Use findings responsibly and follow responsible disclosure practices.

This scanner identifies GitHub Actions workflows that are vulnerable to the **PwnRequest** attack, which occurs when:

1. A workflow uses the `pull_request_target` trigger (runs with write permissions and secrets)
2. It checks out untrusted PR code (`head.ref`, `head.sha`, or `merge_commit_sha`)
3. It executes code from that checkout (npm, make, pip, docker build, etc.)

An attacker can fork the repository, modify build files to exfiltrate secrets, and open a PR to trigger the vulnerable workflow.

TIP: Read the https://securitylab.github.com/research/github-actions-preventing-pwn-requests/[official research] for detailed threat modeling.

[[attack-flow]]
=== Attack Flow Diagram

----
Attacker forks repo
       |
       v
Modifies package.json / Makefile
       |
       v
Opens PR to target repo
       |
       v
Trigger: pull_request_target (with secrets!)
       |
       v
Checkout: ${{ github.event.pull_request.head.sha }} (attacker code)
       |
       v
Execute: npm install / make build (runs attacker code)
       |
       v
Exfiltrate secrets to attacker server
----

[[quick-start]]
== Quick Start

Get started in 4 commands:

[source,bash]
----
# 1. Create input file with GitHub repo URLs (one per line)
echo "https://github.com/your-org/your-repo" > repos.txt

# 2. Clone .github folders from all repos
uv run scripts/clone_repos.py repos.txt

# 3. Scan for vulnerabilities
uv run scripts/scan_repos_async.py

# 4. Review results
cat vulnerabilities.csv
----

TIP: Use `just scan` for the complete scanning workflow.

[[phase1]]
=== Phase 1: Repository Discovery

Find repositories in GitHub organizations that use `pull_request_target`:

[source,bash]
----
# Scan GitHub organizations for repos with pull_request_target workflows
uv run scripts/scraper.py --orgs org1 org2 org3 > repos.txt

# Or scan from a file of organization names
uv run scripts/scraper.py --orgs-file orgs.txt > repos.txt

# Filter for specific repositories (e.g., only production)
uv run scripts/scraper.py --orgs org1 --archived false --fork false > repos.txt
----

**Output:** `repos.txt` - List of GitHub repository URLs (one per line)

[source,bash]
----
https://github.com/redhat-developer/rhdh
https://github.com/openshift/kubernetes
https://github.com/konflux-ci/konflux-ui
----

NOTE: The scraper uses GitHub GraphQL API and requires `gh` CLI authentication.

[[phase2]]
=== Phase 2: Clone Workflow Files

Sparse-clone only the `.github/` folder from each repository for efficiency:

[source,bash]
----
# Clone all repos (uses sparse-checkout for efficiency)
uv run scripts/clone_repos.py repos.txt

# Limit to first N repos for testing
uv run scripts/clone_repos.py repos.txt --limit 50

# Specify output directory
uv run scripts/clone_repos.py repos.txt --output-dir scan-output.txt

# Control parallelism (default: 16 workers)
uv run scripts/clone_repos.py repos.txt --workers 32
----

**Output:** `scan-output.txt/{org}/{repo}/{branch}/.github/workflows/*.yml`

The scanner clones each branch as a separate directory to analyze all workflow versions.

WARNING: Cloning large numbers of repositories can consume significant disk space and network bandwidth.

[[phase3]]
=== Phase 3: Vulnerability Scanning

Scan cloned workflows for PwnRequest vulnerabilities:

[source,bash]

# Full scan (recommended - async for performance)
uv run scripts/scan_repos_async.py

# Refresh git worktrees before scanning
uv run scripts/scan_repos_async.py --refresh-worktrees

# Limit scan for testing
uv run scripts/scan_repos_async.py --limit 100

# Using justfile
just scan

# Scan specific repo only
uv run scripts/scan_repos_async.py --repo redhat-developer/rhdh
----

**Output:** `vulnerabilities.csv`

[[detection-logic]]
== Detection Logic

The scanner (`pwn_request_detector.py`) looks for three conditions in the same job:

=== 1. Trigger: pull_request_target

[source,yaml]
----
on:
  pull_request_target:
    types: [opened, synchronize, reopened]
----

WARNING: This trigger runs with repository secrets and write permissions, even from forked PRs.

=== 2. Checkout: Untrusted PR Code

[source,yaml]
----
- uses: actions/checkout@v4
  with:
    ref: ${{ github.event.pull_request.head.sha }}  # DANGEROUS
----

**Dangerous refs:**

[cols="1,2"]
|===
|Ref |Risk

|`github.event.pull_request.head.sha`
|Checks out attacker's commit

|`github.event.pull_request.head.ref`
|Checks out attacker's branch

|`github.event.pull_request.merge_commit_sha`
|Checks out merge commit

|`github.head_ref`
|Alternative syntax for head.ref
|===

NOTE: Even if you validate the ref later, the checkout has already fetched untrusted code.

=== 3. Execute: Code from Checkout

[source,yaml]
----
- run: npm install  # Executes attacker-controlled package.json
- run: make build   # Executes attacker-controlled Makefile
- uses: ./action    # Executes local action from checkout
----

**Dangerous commands include:**

* Package managers: `npm`, `yarn`, `pnpm`, `pip`, `poetry`, `cargo`, `go`
* Build tools: `make`, `gradle`, `mvn`, `bazel`, `cmake`
* Docker: `docker build`, `docker-compose`
* Scripts: `./script.sh`, `bash script`, `python script.py`
* Actions: `./action`, `actions/github-script` with unsafe inputs

CAUTION: Any command that reads files from the checkout is potentially dangerous.

[[remediation]]
== Remediation

=== Option 1: Use workflow_run (Recommended)

Split into two workflows:

.Before (Vulnerable)
[source,yaml]
----
on: pull_request_target
jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.pull_request.head.sha }}
      - run: npm install
      - run: npm test
      - run: npm run deploy
        env:
          DEPLOY_KEY: ${{ secrets.DEPLOY_KEY }}  # STOLEN!
----

.After (Secure)
[source,yaml]
----
# .github/workflows/pr-build.yml (runs on PR with limited permissions)
on: pull_request
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install
      - run: npm test

# .github/workflows/pr-deploy.yml (runs after build completes)
on:
  workflow_run:
    workflows: ["PR Build"]
    types: [completed]
jobs:
  deploy:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.workflow_run.head_sha }}
      - run: npm run deploy
        env:
          DEPLOY_KEY: ${{ secrets.DEPLOY_KEY }}  # SAFE - only runs on successful build
----

TIP: The build workflow has no access to secrets. The deploy workflow only runs after the build succeeds.

=== Option 2: Only Checkout Trusted Code

[source,yaml]
----
on: pull_request_target
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.pull_request.base.sha }}  # Base branch, not PR
      - run: npm install
      - run: npm test
----

WARNING: This doesn't test the PR changes, only the base branch.

=== Option 3: Require Labels for Execution

[source,yaml]
----
jobs:
  build:
    if: contains(github.event.pull_request.labels.*.name, 'safe-to-test')
    steps:
      # Only runs if maintainer added 'safe-to-test' label
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.pull_request.head.sha }}
      - run: npm test
----

NOTE: This is still a social engineering vector but reduces risk.

=== Troubleshooting

[details="Common Issues"]
--
**Issue:** `gh: authentication failed`

Solution:
[source,bash]
----
gh auth login
# Follow prompts to authenticate
----

**Issue:** `Permission denied: './analyze_repos.sh'`

Solution:
[source,bash]
----
chmod +x analyze_repos.sh
----

**Issue:** `uv: command not found`

Solution:
[source,bash]
----
# Install uv
curl -LsSf https://astral.sh/uv/install.sh | sh
----

**Issue:** Scanner finds no vulnerabilities but you know they exist

Solution:
[source,bash]
----
# Check if workflows were cloned
ls scan-output.txt/

# Refresh git worktrees
uv run scripts/scan_repos_async.py --refresh-worktrees

# Check scanner logs
uv run scripts/scan_repos_async.py --verbose
----

**Issue:** Memory error when scanning many repos

Solution:
[source,bash]
----
# Reduce worker count
uv run scripts/scan_repos_async.py --workers 4

# Scan in batches
head -100 repos.txt > repos-batch1.txt
uv run scripts/scan_repos_async.py --input repos-batch1.txt
----
--
